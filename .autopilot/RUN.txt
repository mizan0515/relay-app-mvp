Core loop summary (informative — authoritative version is PROMPT.md):
- Read .autopilot/STATE.md, then PROMPT.md boot sections in order.
- Run boot: HALT check, LOCK, reschedule watchdog, doctor, state reads,
  probation check, mode decision.
- Active mode → one commit-worthy slice → PR → auto-merge (when allowed) →
  post-merge cleanup (fetch --prune, delete :gone locals, tripwire
  surviving origin/dev/* branches).
- Idle-upkeep mode → repo scan + web search → append ≤10 lines to
  FINDINGS.md. Streak-collapse rule: no-delta iters update an in-place
  counter instead of appending a fresh dashboard/HISTORY row.
- Exit-contract: write METRICS line, remove LOCK, write NEXT_DELAY,
  self-reschedule via ScheduleWakeup tool call + 2-line LAST_RESCHEDULE
  sentinel (line 2 = raw tool response).

Operator signals (preferred paths, single surface each):
- Direction / product decisions → decision PR against base: (loop opens,
  operator only merges). Never ask the operator to edit STATE.md by hand.
- Emergency stop → operator creates .autopilot/HALT via dashboard button
  or `touch .autopilot/HALT`. Loop halts at next boot.
- Resume → operator deletes HALT (or merges a decision PR that resolves
  to `noop` / directive — loop auto-clears decision-related HALTs).

Runner reuse:
- The default runner reuses ONE detached automation worktree (for example
  ..\<repo>-autopilot-runner\live) so the parent directory does not fill
  up with endless iter-* folders. If you need multiple concurrent runners,
  point AUTOPILOT_WORKTREE at distinct, named worktrees — do NOT create a
  new worktree per iteration.
